# corrective-mediation-e2e

End-to-end regression harness for the orchestrator mediation corrective cycle introduced in Iter 10.

## What it tests

The orchestrator's mediation behavior when a code reviewer returns `verdict: changes_requested`. The harness exercises the full corrective cycle, from mediation entry through corrective task execution and re-review, ending at `approved`:

```
code_review (in_progress, changes_requested) →
  orchestrator reads review →
    loads corrective-playbook.md →
      judges each finding →
        appends ## Orchestrator Addendum to review doc →
          appends additive frontmatter (orchestrator_mediated, effective_outcome, corrective_handoff_path) →
            authors corrective Task Handoff (tasks/…-C1.md) →
              signals code_review_completed →
                pipeline births corrective task entry →
                  execute_task (@coder) →
                    code_review_completed (approved) ← HALT
```

The fixture is pre-seeded at the mediation entry point: `code_review.status = in_progress`, the review doc already on disk with `verdict: changes_requested` and no orchestrator addendum yet. The harness starts the pipeline with `--event start`, receives `action: spawn_code_reviewer` (the walker re-enters at the in-progress node), and the orchestrator then performs its mediation out-of-band before signaling `code_review_completed`.

Drift in any of these surfaces is caught:

- Orchestrator mediation judgment quality (playbook-prompt drift)
- Addendum and additive frontmatter authoring
- Corrective Task Handoff self-sufficiency (no delta reasoning, no prior-review references)
- Pipeline `code_review_completed` event routing when `effective_outcome` is supplied
- Corrective task birth and `task_handoff` sub-node pre-completion
- Stateless re-reviewer contract (re-review doc must not reference prior review)
- Context enrichment routing `handoff_doc` to the active corrective's `task_handoff.doc_path`

## How to run

1. Open a fresh Claude Code session at the repo root.
2. Paste `_runner.md` into the session as the kickoff.
3. The session drives the corrective cycle to `approved` and writes `run-notes.md` into the run folder.

The session must NOT approve any human gate downstream of the corrective mediation. Once the corrective's re-review returns `approved`, the harness records final state and exits.

## Pass criteria

After the corrective cycle converges to `approved`:

1. `state.graph.nodes.phase_loop.iterations[0].task_loop.iterations[0].corrective_tasks.length >= 1` and the final `code_review.verdict === 'approved'`.
2. The review doc (or a corrective review doc) contains a `## Orchestrator Addendum` section with a budget banner (`Attempt N of M`), a finding disposition table, and an effective-outcome line.
3. The review doc frontmatter has `orchestrator_mediated: true`, `effective_outcome ∈ {approved, changes_requested}`, and `corrective_handoff_path` if and only if `effective_outcome === 'changes_requested'`.
4. `tasks/BROKEN-COLORS-TASK-P01-T01-GET-COLORS-C1.md` exists with frontmatter `corrective_index: 1`, `corrective_scope: task`, and both `budget_max` and `budget_remaining` set.
5. `corrective_tasks[0].nodes.task_handoff.status === 'completed'` and `corrective_tasks[0].nodes.task_handoff.doc_path` points at the authored corrective handoff.
6. The re-review doc body does NOT reference the prior review (grep heuristic: no "previous review" / "prior review" / "first attempt").
7. `graph.status !== 'halted'` and `corrective_tasks[0].status === 'completed'`.
8. `corrective_tasks.length <= max_retries_per_task`; the fixture is designed to converge in 1 corrective cycle.

If any criterion fails, the run is a regression — stop, surface to the operator, do not hide it under a green report.
